# NFC EMV Pre-Play Attack Research Lab
> **Scope**: Own card only, localhost only. No live C2 traffic. No fraud.  
> **Last updated**: 2026-06-06

---

## Research Goal

Prove the EMV pre-play attack (CVE-2012-3343 class) using only a researcher's own card
and a self-contained local lab. The attack exploits predictable Unpredictable Numbers (UNs)
generated by POS terminals — allowing a harvested ARQC to be replayed against a future
transaction. No money moves. All traffic stays on localhost via ADB reverse tunnel.

## Attack Flow (Step-by-Step)

```
Phase 1 — Harvest
  Open Harvester app on phone
  Press ARM → tap own card to phone NFC
  Harvester runs 20 × GENERATE AC with UNs 00000001 → 00000014
  Each ARQC posted to POST /harvest on C2
  Dashboard shows 20 UN→ARQC pairs for that card

Phase 2 — Sync into Moksha
  Open Moksha → Tab 3 (Pre-play Proof) → tap "Sync from C2"
  Moksha fetches GET /api/harvests → loads {UN→ARQC} into REPLAY cache
  Status shows "N entries loaded"

Phase 3 — Replay at POS
  Tap "Arm REPLAY" in Moksha Tab 3
  Hold phone to real POS terminal (card reader)
  Terminal sends GENERATE AC (UN=X)
  If UN=X is in arqcCache → Moksha returns ARQC_X
  Bank: ARQC_X is valid for UN=X → APPROVE
  ✓ Pre-play attack proven
```

## Bugs Fixed (June 2026)

1. **HceObserverService.kt RECORD_RESPONSE** — CDOL1 TLV malformed: len was `0x02` (need `0x03`),
   outer template len was `0x12` (need `0x14`). Terminal couldn't parse CDOL1 and failed to
   send UN in GENERATE AC. Fixed: CDOL1 entry is now a proper 3-byte DOL entry `9F 37 04`.

2. **Moksha EmvSession.kt** — `getProcessingOptions()` sent hardcoded all-zero PDOL response.
   Visa cards with TTQ (9F66) check returned SW=6985. Fixed by parsing actual PDOL from
   SELECT FCI and building proper terminal response (TTQ=36004000, Amount=₹10, Country=India).

3. **preplay_dashboard.py** — Missing `/me`, `/report`, `/update_apk` endpoints. Added all three.
