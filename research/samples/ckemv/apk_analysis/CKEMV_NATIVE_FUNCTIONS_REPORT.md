# CKEMV-Harvester — Native Functions Report (Operator-Authored Code)

**Audience:** Defender / security researcher familiar with EMV at a high level.
**Sample:** `samples/ckemv/sample.apk` · `xyz.happify.ckemv` · "Geometry Saver".
**Scope:** This report enumerates every function in the two native libraries whose source belongs to the operator (crates `ckemv_android`, `ckemv_emv_kernel`, `ckemv_core`, `ckemv_reporter`). Third-party (open-source) crates linked into the binaries are catalogued separately in `CKEMV_OSS_MODULE_INVENTORY.md` — those are *not* re-derived here.

**Evidence base:**
- `kernel_functions.json`, `reporter_functions.json` — per-function disassembly (offset, size, instruction count, BL-targets resolved to symbols, ADRP/ADD-pair-resolved `.rodata` cross-refs). Generated by `native_analyze.py` (lief + capstone).
- `kernel_operator_functions.md`, `reporter_operator_functions.md` — operator-only subset.
- `kernel_rodata.txt`, `reporter_rodata.txt` — printable `.rodata` strings.
- `kernel_symtab.txt`, `reporter_symtab.txt` — full local symbol tables.
- `samples/ckemv/ckemv_reporter_strings.txt` — 7,291-line strings dump of the reporter.

**Notation in function lists:**
- Offsets are virtual addresses into the `.text` section of the respective `.so`.
- "calls X" means a `BL` instruction whose target was resolved to symbol `X`.
- "→" indicates a direct dependency (call or write).

## Part A — `libckemv_android.so` (Rust EMV kernel)

### A.0 — The JNI bootstrap

#### `JNI_OnLoad` @ `0x496fc` (8,384 bytes; the only function exported via `.dynsym`)

**What it does.** Called once when `System.loadLibrary("ckemv_android")` is executed by `EmvKernel.<clinit>`. It locates the Java class `xyz/happify/ckemv/EmvKernel`, then calls `JNIEnv::register_native_methods` (an `jni`-crate helper) ten times, one per Java native method — wiring each Java declaration to the corresponding Rust `native_*` function. Because the wiring is done via `RegisterNatives` at runtime instead of via the `Java_*` symbol name convention, none of the ten handler addresses appear in the dynamic symbol table; only `JNI_OnLoad` does. This is a defender-relevant defence-evasion pattern: a `nm` of the stripped `.dynsym` reveals essentially nothing about the kernel.

The Java FQCN, method names, and JNI type signatures consumed here are XOR-obfuscated in `.rodata` via the `obfstr` crate. Strings appear in clear only on the stack at runtime, immediately before the `RegisterNatives` call.

### A.1 — `ckemv_android::*` (the 10 JNI handlers)

Each handler follows the same skeleton:
1. Acquire the global `Mutex<Option<TransactionDispatcher>>` via `once_cell::imp::OnceCell::initialize`.
2. If the handler needs an input string, call `JNIEnv::get_string(...)` and convert via `JavaStr → String`.
3. Forward to the appropriate `ckemv_emv_kernel::dispatcher::TransactionDispatcher::*` method.
4. If a string must be returned, call `JNIEnv::new_string(...)`.
5. On any error, return either an empty `String` or `"NotInitialized"`.

#### `native_initialize` @ `0x43a64` (2,168 B, 542 instr, 55 calls, 4 string refs)

**Java signature.** `EmvKernel.initialize(boolean isContactless, String paramsJson) : void`

**What it does.** Wires up the global `TransactionDispatcher`, reads the JSON config from Java, parses it via `serde_json::de::from_trait`, and constructs a fresh `TransactionContext` filled with per-stage TLV overlays. It also installs the Android `log` adapter on first call:

- `android_logger::Config::with_tag("CKEMV_NATIVE")`
- `android_logger::init_once`

Strings it references in `.rodata`:

> `"CKEMV_NATIVE"` (logcat tag)
> `"Initialize: is_contactless=, deterministic_un=, params="` (info log line, with fields substituted at runtime)
> `"ckemv-android/src/lib.rs"` (panic-location backref)
> `"Invalid string"`, `"Error: Invalid APDU Response"`, `"NotInitialized"` (error returns)
> `"fci"`, `"gpo"`, `"sfi"`, `"cdol1"`, `"cdol2"`, `"_":"` (JSON keys expected in `params`)

The presence of `"deterministic_un="` confirms — without dispute — that the kernel supports an explicit *deterministic Unpredictable Number* mode in which `9F37` is supplied from outside instead of generated randomly. This is the textbook pre-play / relay-research toggle.

The `params` JSON keys (`fci`, `gpo`, `sfi`, `cdol1`, `cdol2`) correspond exactly to the per-stage overlay maps that the Kotlin layer flattens from `CveConfig.mastercard_tags` / `visa_tags`.

#### `native_get_next_command` @ `0x442dc` (1,056 B, 264 instr, 28 calls)

**Java signature.** `EmvKernel.getNextCommand() : String`

**What it does.** Locks the dispatcher, calls `TransactionDispatcher::next_command`, formats the returned `ApduCommand` to bytes (`ckemv_core::command::ApduCommand::to_bytes`), and hex-encodes them using the `"0123456789ABCDEF"` table at `.rodata 0x19810`. Returns the hex APDU string to Java. The Java side feeds this string straight into `IsoDep.transceive(byte[] ...)` after `hex.decode`.

#### `native_process_response` @ `0x446fc` (1,708 B, 427 instr, 45 calls)

**Java signature.** `EmvKernel.processResponse(String hexResponse) : void`

**What it does.** Decodes the hex string from Java back to bytes (`<Vec<u8> as hex::FromHex>::from_hex`), wraps them via `ckemv_core::response::Response::from_bytes`, and feeds the response to `TransactionDispatcher::process_response`. Triggers per-Tlv parsing inside the kernel.

Notable rodata: same `"Invalid APDU Response"`, `"NotInitialized"`, and JSON-key constants as `native_initialize`.

#### `native_get_status` @ `0x44da8` (808 B, 202 instr)

**Java signature.** `EmvKernel.getStatus() : String`

**What it does.** Reads the dispatcher's status enum (`InProgress`, `Completed`, `Failed`, `NotInitialized`) and renders it via `alloc::fmt::format::format_inner`. The Java NFC loop (`B2/q.java`) treats the return string `"InProgress"` as the continuation signal.

#### `native_get_pan` @ `0x450d0` (2,816 B, 704 instr, 75 calls)

**Java signature.** `EmvKernel.getPan() : String`

**What it does.** Calls `TransactionDispatcher::get_results`, then reads the `pan` field of the resulting `EmvResults` struct. The unusually high call count (75) is dominated by drop-glue for `EmvResults` and its child `Vec<(u8, u8, String)>` items — this is Rust's `Drop` book-keeping, not business logic.

The five getters below (`get_atc`, `get_arqc`, `get_aid`, plus `get_pan` above) are structurally identical — same size (2,816 B), same instruction count (704), same call list. They differ only in which `EmvResults` field they pluck and stringify before returning.

#### `native_get_atc` @ `0x45bd0` (2,816 B)
#### `native_get_arqc` @ `0x466d0` (2,816 B)
#### `native_get_aid` @ `0x471d0` (2,816 B)

(See `native_get_pan` for the shape; only the field selector differs.)

#### `native_get_tx_parameters` @ `0x47cd0` (3,824 B, 956 instr, 93 calls)

**Java signature.** `EmvKernel.getTxParameters() : String`

**What it does.** Builds a JSON-stringified map of the parameters fed to the chip on the most recent transaction. Internally it grabs both `get_context` (the `TransactionContext`) and `get_results`, then iterates a `BTreeMap` and serializes via `serde_core::ser::Serializer::collect_map` → `serde_json::value::Value::Display`. Hex-string conversion uses the same `"0123456789ABCDEF"` table.

Notable EMV tag literals referenced: `9F02`, `9F21`, `9F32`, `9F36`, `9F37`, `9F48`, `5F24`, `9F46`, `5F34`, `9F47` — these are the names of the EMV TLV tags that the kernel injects into the request to the chip.

#### `native_get_raw_data` @ `0x48bc0` (2,876 B, 719 instr, 56 calls)

**Java signature.** `EmvKernel.getRawData() : String`

**What it does.** Stringifies the *entire* TLV tree (`EmvResults.raw_tlv_tree`) into a JSON object with TLV tag → value pairs. Uses `alloc::str::join_generic_copy` and `alloc::raw_vec::RawVec::grow_one` to stitch nested levels together. The result is what the Java side stores in `CardRecord.raw_data` and uploads to the C2.

### A.2 — `ckemv_emv_kernel::dispatcher` (the orchestrator)

The dispatcher decides which kernel strategy (Visa vs Mastercard) to run, advances it, and reports state. Owned by a global `Mutex<Option<TransactionDispatcher>>`.

#### `TransactionDispatcher::get_context` @ `0x62590` (184 B)
**What it does.** Returns a clone of the current `TransactionContext` (clones the embedded `hashbrown` HashMap of TLV overlays).

#### `TransactionDispatcher::status` @ `0x62648` (156 B, 1 string ref)
**What it does.** Returns one of `NotInitialized` / `InProgress` / `Completed` / `Failed` as a `String`. The rodata window also contains `"Dispatcher failed before kernel activation"` and the PPSE/PSE constants `315041592E5359532E4444463031` (`"1PAY.SYS.DDF01"`, contact PSE) and `325041592E5359532E4444463031` (`"2PAY.SYS.DDF01"`, contactless PPSE).

#### `TransactionDispatcher::get_results` @ `0x626e4` (372 B)
**What it does.** Returns the `EmvResults` struct. Falls back to `EmvResults::fallback_track2` if direct PAN extraction failed (extracts the PAN from the Track 2 equivalent data instead — tag `57`).

#### `TransactionDispatcher::next_command` @ `0x62858` (1,336 B, 334 instr, 17 calls, 5 string refs)
**What it does.** Drives the protocol state machine. Returns either a "SELECT PPSE" APDU (first call), a "SELECT AID" APDU (after PPSE response), or delegates to the chosen kernel's `next_command`. Falls back to the hard-coded Visa AID `A0000000031010` if no PPSE entries match.

Notable rodata strings (state-machine log lines):

> `"Selecting PPSE/PSE ()"`
> `"Selecting Application AID: "`
> `"No supported applications found on this card"`
> `"PPSE selected, moving to AID selection"`
> `"PPSE failed, falling back to default AID"`
> `"Unsupported AID: "`
> `"Dispatcher processed response"`
> `"APDU Error: "`

#### `TransactionDispatcher::process_response` @ `0x62d90` (2,572 B, 643 instr, 56 calls)
**What it does.** Receives a chip response, parses TLVs via `iso7816_tlv::ber::tlv::Tlv::parse_all`, classifies the scheme (Visa vs Mastercard), allocates the appropriate kernel (`VisaKernel` or `MastercardKernel`) on the heap via `alloc::alloc::exchange_malloc`, and forwards subsequent responses to that kernel's `process_response`. The decision happens by inspecting the AID prefix from the PPSE entry.

#### `TransactionDispatcher::extract_aids_from_ppse` @ `0x6379c` (292 B, 11 calls)
**What it does.** Pulls the BER-TLV from a PPSE response, walks the directory entries, and returns a `Vec<String>` of AID hex strings.

#### `TransactionDispatcher::find_aids_recursive` @ `0x638c0` (280 B, 6 calls)
**What it does.** Inner helper for `extract_aids_from_ppse` — recurses through nested BER constructed objects looking for tag `4F` (ADF Name / AID).

### A.3 — `ckemv_emv_kernel::kernels::visa` (the Visa kernel)

#### `<VisaKernel as EmvKernelStrategy>::next_command` @ `0x65a04` (1,104 B, 276 instr, 22 calls)
**What it does.** Produces the next APDU based on the kernel's internal `VisaState`. Five state-machine log strings give away the flow:

> `"Requesting Processing Options (GPO)"`
> `"Reading Record (SFI: , Record: )"`
> `"Requesting 1st Generate AC (ARQC/TC)"`
> `"Selected Visa Application"`
> `"GPO returned ARQC. Visa Fast-Path completed."`
> `"GPO returned ARQC and no records to read. Complete."`
> `"GPO successful. No records to read."`
> `"GPO successful. Found  records to read."`
> `"GPO successful (no AFL). Waiting for GenAC."`

Calls into `ckemv_emv_kernel::dol::build_dol_payload` for PDOL / CDOL1 assembly.

The **"Visa Fast-Path"** branch is the well-known quirk where the cryptogram is returned directly in the GPO response template (`9F26` inside `77`) instead of needing a separate GENERATE AC step — this is the qVSDC path. The kernel handles both qVSDC and the slow VSDC path explicitly.

#### `<VisaKernel as EmvKernelStrategy>::process_response` @ `0x65e54` (3,940 B, 985 instr, 68 calls, 23 string refs)
**What it does.** Parses each card response, dispatches by current state, drives the SDA/DDA verification at the right step (calls `ckemv_emv_kernel::crypto::oda::verify_certificates`), and on the final GENERATE AC response harvests the ARQC.

Notable log strings:

> `"Did not receive cryptogram in Generate AC response"`
> `"Card returned error status: "`
> `"Read record / "`
> `"All records read. Fast-Path ARQC already captured. Complete."`
> `"All records read. Proceeding to GenAC."`
> `"ODA Verification Failed: "`
> `"Successfully obtained cryptogram or track2 data."`
> `"Invalid AFL record range"`, `"AFL length must be a multiple of 4"`, `"Invalid AFL hex: "`

#### `<VisaKernel as EmvKernelStrategy>::status` @ `0x66db8` (68 B)
Returns the kernel's status enum.

#### `<VisaKernel as EmvKernelStrategy>::get_context` @ `0x66dfc` (152 B)
Returns a clone of the context (`hashbrown::RawTable::clone`).

#### `<VisaKernel as EmvKernelStrategy>::get_results` @ `0x66e94` (3,448 B, 862 instr)
Clones the populated `EmvResults` struct for the caller. The size is dominated by deep `Clone` impls for every field.

### A.5 — `ckemv_emv_kernel::dol` (Data Object List builder)

#### `build_dol_payload` @ `0x67c88` (6,212 B, 1,553 instr, 118 calls)

**What it does.** Given a DOL (PDOL or CDOL) byte stream that the card returned (a packed list of `(tag, length)` pairs the card wants supplied), this function walks the list, looks up each tag in the current `TransactionContext.tags` HashMap, supplies the value (truncated/padded to `length`), and concatenates the result. If a tag is not in the context map, it falls back to a tag-specific default — that's the "negative correlated time" path for `9F37` (the Unpredictable Number) and a current-date default for `9A`.

The kernel's per-tag fallback table embeds the following EMV tag literals (from `.rodata 0x19b60–0x19bec`):

`5F24` (Application Expiration Date) · `5F34` (PAN Sequence Number) · `9F02` (Amount, Authorized) · `9F21` (Transaction Time) · `9F32` (Issuer Public Key Exponent) · `9F36` (ATC) · `9F37` (Unpredictable Number) · `9F46` (ICC Public Key Certificate) · `9F47` (ICC Public Key Exponent) · `9F48` (ICC Public Key Remainder).

The fact that fallback values exist for these specific tags (not just *any* EMV tag) is informative — these are the fields the operator wants the kernel to autofill if the Java config didn't override them. `9F37` and `9A` autofill turns the harvester into a "always-produces-a-fresh-cryptogram" device even without a fully spelled-out scenario.

### A.6 — `ckemv_emv_kernel::crypto::oda` (Offline Data Authentication / RSA)

EMV ODA — Static Data Authentication (SDA), Dynamic Data Authentication (DDA), and Combined Data Authentication (CDA) — uses operator-issued RSA keys. The card carries an Issuer PK certificate (signed by the scheme CA) and an ICC PK certificate (signed by the issuer). The kernel verifies both before trusting subsequent cryptographic data.

Importantly the kernel **does not need to do this** for cryptogram capture to succeed — the ARQC is produced by the chip regardless of whether the terminal verified ODA. The fact that the operator implemented full ODA suggests the kernel is shared between the *harvester* role and a separate (possibly PoS-side) role where ODA matters.

#### `recover_cert` @ `0x695a8` (2,416 B, 604 instr, 37 calls)
**What it does.** Recovers a certificate's plaintext from RSA-encrypted bytes using `num_bigint::BigUint::modpow` (RSA decrypt is `c^e mod n` for a public exponent e). Bidirectional bit/digit conversions are done via `num_bigint::biguint::convert::from_bitwise_digits_le` and `to_bitwise_digits_le`.

#### `verify_issuer_public_key` @ `0x69f18` (1,728 B, 432 instr, 36 calls)
**What it does.** Verifies the recovered issuer certificate by reconstructing the static-data digest, hashing with `sha1::compress::compress`, and comparing. Raises `"Issuer PK Certificate Hash mismatch"` on failure.

#### `verify_icc_public_key` @ `0x6a5d8` (1,760 B, 440 instr, 38 calls)
**What it does.** Same flow for the ICC certificate. Raises `"ICC PK Certificate Hash mismatch"` on failure. Pulls the AID (`9F06` / `4F`) to look up the right CA root.

#### `build_static_data` @ `0x6acb8` (1,392 B, 348 instr, 18 calls)
**What it does.** Concatenates the byte ranges of the records identified by the AFL (Application File Locator) as input to the SDA hash. Records 1..N from the SFI list, in order.

#### `verify_certificates` @ `0x6b228` (2,624 B, 656 instr, 56 calls, 11 string refs)
**What it does.** The orchestration entry point — calls `recover_cert` → `verify_issuer_public_key` → `build_static_data` → `verify_icc_public_key` in sequence. Logs `"Issuer PK Verification Failed: "` or `"ICC PK Verification Failed: "` with the underlying error. Error vocabulary in rodata:

> `"No AID found"`, `"Invalid AID hex"`, `"AID too short to extract RID"`
> `"No CA PK Index found"`, `"Invalid CA PK Index"`, `"CA Public Key not found"`
> `"No Issuer PK Cert found"`, `"Invalid Issuer PK Cert hex"`
> `"Invalid ICC PK Cert hex"`, `"Missing Issuer Remainder"`, `"Missing ICC Remainder"`
> `"Recovered length does not match modulus length"`
> `"Invalid Issuer Cert format"`, `"Invalid ICC Cert format"`

#### `verify_certificates::_{{closure}}` @ `0x6bc68` and `0x6bcf0` (136 B each)
Closure helpers used inside `verify_certificates` — they're the closures passed to `Result::map_err` to wrap the underlying error string.

The kernel **bundles a CA-key provider trait (`dyn CaKeyProvider`)** but the implementation is not in `verify_certificates` itself — it's a trait object stored in the kernel's context. The keys come from a Rust constant table elsewhere (consult `_<ckemv_emv_kernel::crypto::CaKeyProvider as ...>` impls — these are the `core::ptr::drop_in_place<ckemv_emv_kernel::crypto::PublicKey>` references in the dispatcher).

### A.7 — `ckemv_emv_kernel::tags` (EMV tag-name lookup)

### A.8 — `ckemv_emv_kernel::results` (TLV → JSON-ish tree)

#### `EmvResults::fallback_track2` @ `0x63bec` (644 B, 161 instr)
**What it does.** If the kernel didn't find tag `5A` (PAN) directly, extracts the PAN from the Track 2 equivalent data (tag `57`) by splitting on the field-separator byte (`0xD`).

#### `build_tlv_tree` @ `0x63e70` (1,228 B, 307 instr, 33 calls)
**What it does.** Recursively walks a parsed list of `iso7816_tlv::ber::tlv::Tlv` objects, calls `ckemv_emv_kernel::tags::get_tag_name` for each, and serializes them into a `ParsedTlv` tree. Recurses on constructed (`is_constructed`) tags.

#### `extract_emv_data` @ `0x6433c` (5,216 B, 1,304 instr, 83 calls, 10 string refs)
**What it does.** The biggest single function in the kernel. Walks the response TLV tree, pulls out the operationally interesting fields — PAN, AID, ATC, ARQC, Track 2, CDOL1, CDOL2, expiration, etc. — and populates the `EmvResults` struct. Decoders for compact numeric tag formats live here.

### A.9 — `ckemv_emv_kernel::types`

#### `generate_negative_correlated_time` @ `0x61538` (452 B, 113 instr)
**What it does.** Produces a `9F37` (Unpredictable Number) value that is *negatively correlated* with the current wall-clock time — that is, takes the current monotonic time, hashes/derives a 4-byte value from it. Combined with the explicit `deterministic_un=` flag in `native_initialize`, this is the kernel's deterministic-UN feature: the operator can either let `9F37` be predictable-from-time or override it from the C2 config.

This is a strong relay/pre-play research feature — defenders should regard the presence of this helper alone as a smoking gun.

#### `<TransactionContext as Default>::default` @ `0x616fc` (100 B)
Constructs an empty context.

#### `<TransactionContext as Clone>::clone` @ `0x639d8` (148 B)
Deep-clones the context (clones the embedded `hashbrown` RawTable).

#### `<TransactionStatus as Debug>::fmt` @ `0x439f4` (112 B)
Renders the status enum for log lines — `InProgress` / `Completed` / `Failed` strings live in rodata at `0x19ca7` and adjacent offsets.

### A.10 — `ckemv_core` (shared command/response types)

#### `ckemv_core::command::ApduCommand::to_bytes` @ `0x534a4` (664 B, 166 instr)
Serializes an `ApduCommand { cla, ins, p1, p2, data, le }` struct into the on-the-wire APDU byte format. Calls `bytes::buf::buf_mut::BufMut::put_slice`. This is what `native_get_next_command` returns to Java (after hex-encoding).

#### `ckemv_core::response::Response::from_bytes` @ `0x7a018` (184 B)
Parses a raw response into `Response { data, sw1, sw2 }`. Returns an `Err(...)` if the buffer is shorter than 2 bytes (`"Response too short: must be at least 2 bytes"`).

#### `<ckemv_core::response::status::StatusWord as Debug>::fmt` (two instances @ `0x6592c` and `0x6bf08`)
Debug-renders the status word as `"StatusWord { sw1: 0xXX, sw2: 0xXX }"`. Two copies are emitted because the compiler instantiates one per crate that imports the type (Visa kernel + M/Chip kernel).

## Part B — `libckemv_reporter.so` (Rust C2 client)

The reporter is much simpler than the kernel: only three operator-authored functions, all JNI shims that wrap `ureq`. The bulk of the 3 MB binary is `rustls`, `ring`, `webpki`, `ureq`, `url`, and the embedded Mozilla CA bundle.

#### `JNI_OnLoad` @ (early `.text`, registered via `RegisterNatives`)
Wires the three `xyz.happify.ckemv.Reporter` Java native methods to their Rust handlers. The Java FQCN and method names are `obfstr`-XOR'd in rodata at `0x2b0f6–0x2b253`. As with the kernel, no Rust handler appears in `.dynsym`.

#### `ckemv_reporter::native_upload_harvest_data` @ `0xde7a4` (3,312 B, 828 instr, 76 calls, 3 string refs)

**Java signature.** `Reporter.uploadHarvestData(String apiKey, String url, String json) : String`

**What it does.**
1. Pulls all three Java strings via `JNIEnv::get_string`.
2. Deobfuscates header literal strings via `obfstr::xref::inner` (the `Authorization`, `Content-Type` strings live as XOR blobs; the actual bearer/basic prefix is decoded at this point).
3. Calls `ureq::post(url)`.
4. Adds headers via `Request::set(name, value)` (one of them is the Authorization header sourced from the `apiKey` parameter, plus `Content-Type: application/json`).
5. Calls `Request::send_json` with the parsed `serde_json::Value` of the body.
6. Reads the response via `Response::into_string`.
7. Returns the response string to Java via `JNIEnv::new_string`.

The Java upload loop (`B2/s.java`) treats the return string as success if it contains the literal substring `"Success"`.

## Part C — How the JNI surface is registered (anti-static-analysis pattern)

The operator's choice to use `JNI_OnLoad + RegisterNatives` instead of the conventional `Java_xyz_happify_ckemv_*` symbol naming means:

- `nm libckemv_android.so` (against `.dynsym`) shows **only** `JNI_OnLoad`.
- All 10 handler addresses are findable only via `.symtab` (which the stripper kept here — easily blown away in a future build).
- The Java class/method signature strings consumed by `RegisterNatives` are **`obfstr`-XOR'd** in `.rodata`, so a strings dump of the rodata section does **not** reveal them.

**Defender implication.** A YARA rule that scans for the literal string `"xyz/happify/ckemv/EmvKernel"` will *miss*. Rules need to anchor on something else — e.g. the EMV state-machine log strings (Part A.3 / A.4) or the unique combination of crate names in panic-format strings (`"ckemv-emv-kernel/src/kernels/visa.rs"`, `"ckemv-emv-kernel/src/kernels/mchip.rs"`, `"ckemv-emv-kernel/src/dispatcher.rs"`, `"ckemv-android/src/lib.rs"`, `"ckemv-reporter/src/lib.rs"`). These five panic-location strings are excellent candidates for SIGMA/YARA static signatures because they are byte-identical across rebuilds of the same crate workspace.

## Part D — Defender-relevant takeaways

1. **No payload-layer crypto.** `ring` is consumed exclusively by `rustls` for TLS — no `Ed25519Signer`, `crypto_box`, or HMAC use outside cert paths. The HTTPS connection is the only confidentiality layer. **No pinning** — defenders with TLS-intercepting proxies can fully unwrap traffic.
2. **No anti-debug, no anti-instrumentation.** The entire opsec budget is spent on (a) the masquerade UI, (b) the `obfstr` XOR of JNI signature strings, and (c) skipping `.dynsym` for the handlers. Nothing else.
3. **The kernel exposes a deterministic-Unpredictable-Number mode** (`native_initialize` accepts `deterministic_un=true`). This is a meaningful capability — the operator can replay a chosen `9F37` against a card, e.g. to feed it identical UNs across reads, which has obvious implications for relay/pre-play scenarios.
4. **Full ODA implementation** (SDA/DDA/CDA) exists even though the harvester does not strictly need it. This signals the same crate is used elsewhere in the operator's toolkit (a PoS-side or relay-target component).
5. **EMV tag-name table at `0x1ff76`+** in the kernel binary is a very strong static fingerprint — see Part A.7.
6. **Per-crate panic-location strings** (`ckemv-emv-kernel/src/...`, `ckemv-android/src/lib.rs`, `ckemv-reporter/src/lib.rs`) survive every rebuild as long as the workspace layout is unchanged — robust YARA anchors.
7. **`User-Agent: ureq/2.12.1`** on outbound HTTPS from an Android device is anomalous; combined with TLS 1.3 + ECH + MLKEM768 hybrid + the absence of `okhttp` JA3/JA4 markers, the network fingerprint is distinctive at SIEM level.
